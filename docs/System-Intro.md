# Learning Library · 系统介绍

## 一句话

把任意一个主题（一本书、一套方法论、一个软技能领域），变成一段可以让学员真正"下手做"的旅程。不是读它，是反复执行它指向的那几个核心动作，在情境复杂度逐渐上升的过程里练到能迁移。

## 业务假设：我们为什么存在

我们押了三件事：

1. **学员默认是被动的。** 大多数人打开一份课程是想"看完它"，不是想"做一遍"。任何强迫他被动接收的产品形态都在跟用户的真实意图背道而驰。
2. **常规的互动没意思。** 单选、拖拽、填空在大脑里激活的是识别（recognition），不是生成（generation），认知投入度只有真正下手做的一小部分。这种互动可以被算作"做了练习"，但和习得相距甚远。
3. **能留下的是和学员自身有关的。** 让学员在自己的场景、自己的语境、自己的判断里使用这套方法，学到的东西才能被搬走。

我们做的不是"把知识讲给学员听"，是把"被动 → 主动"这个转换做成软件本身的形状。北极星指标和多邻国一样：他下次还回不回来。

## Gamecore：把主题变成游戏的核心机制

我们造了一个概念叫 Gamecore。它不是任何具体的玩法，是一个结构性的判断：

> 任何值得学的主题，都可以被压缩成一组（≤5 个）需要反复执行的核心动作。学员能不能在越来越复杂的情境里把这几个动作执行得越来越好，是这门课唯一应该被衡量的东西。

围绕这条判断，系统由四个咬合的部件组成：

- **核心动作**：直接指向能力迁移的具体行为。如情境领导力的"读懂下属准备度"、非暴力沟通的"把评价改成观察"。
- **质量矩阵**：每个动作有 2-3 个评估维度（如能力线索、意愿线索、证据），每个维度对应低/中/高三档情境复杂度，每个格子是一份 mini-rubric（好/中/差三级具体描述）。这是评分的唯一依据，LLM 不在裸推理。
- **章节—挑战—脚手架阶梯**：旅程由章节组成，章节由挑战组成，每个挑战内部还有一架"脚手架阶梯"：从轻量的故事选择，到结构化表单，到自由文本，渐进释放学员自己产出的责任。
- **道具**：邮件、档案、清单、会议记录这种学员可以"翻"的物件，按场景需要在对话流里出现，不是开场全堆出来。

设计阶段一次性把这套结构填好，运行时按结构跑。LLM 不需要在线即兴推理整个学习设计。

## 设计阶段：从一句话主题到一条完整旅程

设计师的角色像剧本作者 + 教学设计师。一个 Design Copilot 陪她走五步：

| 环节 | 做什么 | 产出 |
| --- | --- | --- |
| Skill 1 · Gamecore 萃取 | 从主题中提炼核心动作和质量矩阵 | 核心动作清单 + 质量矩阵 + 框架概念（如 R1-R4 准备度档位） |
| Skill 2 · 体验选型 | 给每个动作选一种承载形式 | 形式映射表 |
| Skill 3 · 剧本填充 | 写章节、挑战、人物、道具、脚手架阶梯 | 完整剧本 + 答案库（model_judgment + common_misreadings） |
| Skill 4 · 伴学设计 | 设计可解锁的伴学清单 | 伴学清单 + persona + 解锁/升级路径 |
| Skill 5 · 积分配置 | 校准积分的获取/消耗/衰减参数 | 积分参数表 |

每一步都要设计师确认。后面的步骤可以读前面的，但不能覆盖。所有判断都落在一份结构化数据（Blueprint）里，运行时各角色从这里取自己需要的字段。

设计师在 Skill 3 阶段除了写剧本，还要为每个会被评分的脚手架档（rung）补两件东西：

- **model_judgment**：这道题正确读法是什么、理由是什么，用人物名字和具体细节说话。
- **common_misreadings**：学员可能在这一档犯的几种典型偏读，每条对应一个 rubric 维度。

这两件东西在运行时被用来在学员答错时直接给答案。下文"产品判断"那一节会展开。

## 学习阶段：学员每一回合发生了什么

四层嵌套的反馈循环驱动着旅程，从 200 毫秒级的"这一步做得怎么样"到周级的"我整体进步了多少值得回来吗"。每一层有自己的情绪锚点：

| 循环 | 学员心里问的问题 | 系统回答的方式 |
| --- | --- | --- |
| 执行—反馈 | 我这一步做得怎么样 | Judge 评分 + Narrator 即时叙事 |
| 挑战 | 这个情境我处理得如何 | 积分结算 + 脚手架释放/升档 |
| 章节 | 这段旅程我走到哪了 | 伴学接入 + 章节宣言（Manifesto） |
| 旅程 | 我整体进步多少 | 积分衰减 + 伴学解锁 + 招式卡积累 |

在每一回合里，4 个角色协同：

- **State Manager**（代码模块）：所有确定性计算和状态持久化。积分、解锁、衰减、位置全在这里，不经过 LLM。
- **Judge**（LLM）：评估学员表现 + 路径决策 + 派单。它不写学员可见的字。
- **Narrator**（LLM）：旅程的主持人，把 Judge 的判决译成具体可信的场景叙述。它的视野是收窄的：它看不到分数细节，只接收路由决定，避免把评分语调渗进对学员说的话里。
- **Companion(s)**（LLM）：每种伴学一个独立 Agent，按自己的 persona 说话。Judge 决定本轮谁开口、谁沉默、谁还在伏笔里。

## 我们在产品上做出的几个关键判断

每条都对应一个具体的产品形态选择，背后是一个对学习/游戏/产品的具体看法。

### 该说答案时直接说答案

老问题：传统教学产品在学员答错时倾向"再试一次"或者用反问把他往答案方向推（"你确定吗？""换个角度想想？"）。学员感觉系统明明知道答案却装不知道在拷问他。

我们的判断：学习的四步是尝试、知道对错、理解为什么、内化。卡在第二步和第三步之间的时候，再问一次是负价值。所以系统在三种情况下会主动给答案：

- 学员在单选里选了被设计师标为"明显偏读"的选项 → 立刻给答案。
- 表单或自由文本同档连续两次没过 → 第二次没过立刻给答案。
- 学员主动求助（"揭晓"按钮）→ 给答案，扣点积分作为让步。

答案不是 LLM 当场即兴写的，是 Skill 3 在设计阶段写好的 model_judgment + common_misreadings 组合。Narrator 只负责用 HBR 风格的中文把它说通顺，不负责重新推理。这一段没有问号、没有"你愿意重新对应吗"这种半藏半露的句式，结尾让位给学员自己继续。

### 跨挑战切场不复述身份和目标

老问题：每次切到新挑战，开场再来一遍"你是一家中型设计公司的项目经理，这一段要完成的是…"。学员已经知道这些，重复一遍是噪音，是课程介绍腔调。

我们的判断：旅程级身份（protagonist_role）和旅程目标（journey_goal）只在最开头一次性建立。之后每次切场，开场只做三件事：把上一挑战的动作收个尾、把新场景摆到学员面前、最后抛一个具体问题。Narrator 提示词里把 cross_challenge 变体的硬结构和 first 变体分开，校验层会拒掉任何复述身份/目标 8 字符以上的开场。

### 反 AI 味的提示词风格

每个系统提示词都过一道风格守门：禁止"驾驭""引擎""高瞻远瞩""不仅...更是..."这类 AI 味修辞，禁止"我接住你了"这种安抚套话，禁止以破折号代替逗号的句式。目标是让学员读到的中文像 HBR 编辑写的，不像 ChatGPT 默认输出。

这一条乍看是文风审美问题，实际是产品级判断：学员对"AI 味"的识别灵敏度比想象中高，一旦中招就会不自觉地把你的产品归类成"那种 AI 玩具"，留存断崖。我们把它当一级护栏来守。

### 道具原地展开，不弹窗

学员收到的"新文件"（一份邮件、一张档案、一个清单）默认是一个 clamp 预览，点击在对话流里原地展开成全文，不弹模态框。版本切换和"对比上一版"的能力都内嵌在展开后的气泡里。

判断的依据：学员的阅读和思考是在对话流里发生的，弹窗会粗暴地打断这条流。学员关掉弹窗后还要回头去找上下文。原地展开让所有材料都在他眼睛已经在看的地方。

### 招式卡 = 命名认知动作

学员做出的好的认知动作（如"分任务诊断"，即不给整个人贴标签而是按任务分别评估）会被 Judge 在合适的时刻发"招式卡"：那一刻学员的原话被截下来作为他获得这张卡的"高光时刻"。学员一章累积几张招式卡，章节末尾系统会把他的原话编成一段第一人称的"章节宣言"（Manifesto）作为收束。

这件事的产品意义：把"我学会了什么"从系统判定变成学员自己的话。学员看到的不是评分单，是一段他自己说过的话变成的小作品。

### 伴学不是脚手架

脚手架是学员卡住时系统自动施援的临时认知降负荷工具（worked example、对比案例、概念清单等八种）。
伴学是用积分换的、长期解锁的、有人格的角色：向导、旅伴、竞争者、对手、案例库、复盘视角、隐藏剧情线、难度调节器、情境变体包。

这两件事在产品上严格分开：脚手架是被动得到的帮助，伴学是主动选的奖励。学员要想见到资深的伴学，得先好好学。这是系统里那个游戏循环的核心：练习 → 积分 → 解锁 → 体验更深 → 想要更多 → 回去练习。

### 单机为底，社交是后话

每段旅程默认是一个人独自走完的。剧本里所有伴学接入点都预设了"有伴学版"和"无伴学版"两条分支，一个人也走得通。没有强制的协作、没有匹配机制、没有排行榜。社交体验不是底层结构，是后来可以加的一层。

这个判断降低了上线门槛，也让产品对个体学员更诚实：你来这里是为了你自己。

## 这套架构可以延伸到哪里

主题不可知是这个项目最大的机会。如果设计阶段的 5 个 Skill 足够稳，理论上任何主题都可以装进这条管道：

- **领导力 / 沟通 / 管理**这类软技能：核心动作清晰，复杂度可被情境化，最贴这套结构。
- **专业判断**类（医生问诊、设计师评审、销售客户分析）：核心动作就是"读懂情境 + 给判断 + 解释"，质量矩阵的维度天然存在。
- **入职前 / 转岗前的预热**：不是替代培训，是让学员在正式课开始之前已经在头脑里跑过一遍真实场景。
- **认证与考核**：旅程末态可以作为客观能力证据，比任何选择题考试都更接近"他是不是真的会"。

更大的可能性是：Gamecore + Skills + Runtime 的分层本身可能成为一类软件，把"主题到反复练习的旅程"这件事做成一个工业流程。这条路径下，谁占了"主题转换"这一层，谁就占了下一代 ed-tech 的基础设施位。

## 一句话收尾

在所有教学产品都在比谁的内容更全、谁的题库更大、谁的视频更精的时候，我们打的赌是：内容可以是模型已经知道的，题库可以是程序生成的，视频根本就不是核心；真正稀缺的是把人从被动拉到主动的那一种软件形状。Learning Library 是这种形状的第一版。
