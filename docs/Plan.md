# 7/6c4c Learning Library

领域: 7 教育
上次编辑时间: April 22, 2026 6:26 PM
创建日期: March 28, 2026 2:27 PM

UMU 做的一个项目。初步要求：

- Learning Library，利用基模对书的“了解”做课，“以书（topic）的名义”，重点不是展示书，而是让学习者participate，engage every participate in realtime
- 重点是提供促进思考的脚手架、学习之旅
- 以学习者为真正的中心，让学习者不断的参与、动手动眼动脑，这样就能避免任何侵犯版权的可能

业务思考：

对于 Learning Library, 我有一些业务假设：

- 学员默认是被动的，我们的产品存在的理由是把被动转为主动
- 常规的互动（单选题、拖拽、填空）并没有意思
- 真正能留下的一定适合用户自身相关的

从教学设计层面，我造了一个概念叫做“Gamecore”。我好好想了一下，觉得买Learning Library课程的人可能并不会觉得这个课能带来直接的业务结果，所以其实没准课程面向的就是C端或者直接用软技能的，要么再实用一点就是情境化的课程，甚至偏咨询。而从这个角度，相当于核心在于一个很棒的方法论要如何迁移，是这个项目的重中之重。

在这个背景下，“Gamecore”是来自“梦核”，但核心想传达的是用游戏化无痛的方式创造学习之旅之前，需要对学习主题有一个分析，看其中要传达哪些“干货”，这些干货适合用什么方式传达和组织，然后再看 [沉浸式学习体验设计（Immersive Learning Experience Design）](https://www.notion.so/Immersive-Learning-Experience-Design-da38b311e0424198864ff926411bda4d?pvs=21) 中的哪些形式是否适配。

Gamecore 中，我认为最重要的是“爽感”，真的可以参考多邻国的思路，抓北极星指标，就是“学员有没有回来”。在教学中，游戏设计并不是一个完整的工业流程，而是通过一些特点营造一种游戏感。

在这个意义上， Zimmerman 的“反复执行的动作”切中肯綮。每一个主题都一定能落在一个或一组需要学员反复执行的动作中。这个动作直接就是指向迁移的：只要学员在逐渐变复杂的场景和情况下能执行好这些动作，就意味着学习的正反馈。同时，一个学习旅程中的动作需要只是精选的几个动作，不超过 5 个，然后彼此之间的关系要非常明确。

我认为高级伴学是独立于路径的，因为高级伴学是动态加载的。学习本身可以就是规整的、孤独的，但是不同高级伴学可能给到特别不一样的体验，可以拓展对方爽感的上限和学习的深度。要想体验很牛逼的高级伴学，就必须前面好好学，然后一直通关、一直重复动作，才能获得积分、换取高级伴学。

高级伴学不等于脚手架，脚手架是根据学员的水平直接和路径和动作匹配的，而高级伴学是需要解锁的奖励。

## 业务设计

### 设计阶段：为一个主题生成完整的学习旅程蓝图

整个设计阶段的输入是一个**主题**（一本书、一套方法论、一个技能领域），产出是一份可直接进入学习阶段运行的**旅程蓝图**。

### 环节 1：Gamecore 萃取

- 输入：主题名称（如「情境领导力」「高效能人士的七个习惯」）
- 数据来源：**模型的世界知识**，不依赖具体书籍文本
- 产出：≤5 个核心动作 + 动作之间的关系图 + **每个核心动作的质量矩阵**
- **质量矩阵**结构：评价维度 × 应用复杂度
    - **评价维度**（行）：过程信号（推理路径、信息采集、思维结构）+ 结果信号（判断准确性、产出完整性）。具体维度因核心动作而异，每个动作 2-3 个维度
    - **应用复杂度**（列）：高 / 中 / 低——对应情境的干扰程度、线索模糊度、需权衡的因素数量
    - 每个格子 = 一份 **mini-rubric**：该维度在该复杂度下的**好 / 中 / 差**三级描述。例如「信息采集 × 中等复杂度」的格子：好 = 区分出表面与底层信号并引用具体场景细节；中 = 注意到不一致但未深入分析；差 = 只关注了表面信号
    - LLM 评估时只需读取当前挑战对应复杂度的那一列 mini-rubric，对照学员表现输出各维度的好/中/差等级
    - 积分公式：各维度得分（好/中/差 → 分值）× 复杂度倍率 → 求和 = 该次交互的积分。具体分值和倍率由**通用积分框架**定义（见环节 5）
    - 历史表现可压缩为摘要传入 LLM，降低判断成本
- 核心动作的定义：学员在学习旅程中需要反复执行的、直接指向能力迁移的行为。每次执行的情境复杂度递增，执行质量的提升就是学习的正反馈。
- ⚠️ **Human 必须确认产出**，这是整条链路的北极星

### 环节 2：游戏体验选型

- 输入：环节 1 确认后的核心动作清单（**依赖环节 1**）
- 产出：为每个核心动作选定一种承载形式（What-If 模拟器、教会 NPC、悬疑调查、概念法庭……）
- 选型依据：核心动作的知识类型 + 目标认知参与层级
- ⚠️ **Human 必须确认产出**——游戏形式的选择直接决定后续剧本和伴学的设计空间

### 环节 3：剧本与情节生成（含里程碑、分支路径、高级伴学接入）

- 输入：环节 1 + 2（**依赖环节 2 确认后的游戏形式**——先确认核心动作用什么形式承载，再做剧本设计）
- 产出：一条完整的叙事旅程 + 与之交织的高级伴学接入设计，包含：
    - 主路径 + 分支路径
    - 里程碑（关键检查点）
    - **章节-挑战结构**：旅程由若干章节（Chapter）组成，每章包含若干挑战（Challenge），每个挑战围绕一个或多个核心动作，情境复杂度在章节内和章节间递增
    - 高级伴学接入点 + 接入后的互动变化
- 关键约束：
    - **单人旅程为底层结构**——学员独自也能走完全程
    - 剧本中需预留「伴学接入点」——在特定挑战节点上，如果学员解锁了某类高级伴学，该伴学可以自然加入当前挑战，改变互动方式（而非打断叙事）
    - 剧本不能假设伴学必然在场，也不能假设伴学必然缺席；它需要为两种情况都写好「接口」
- **高级伴学与剧本的游戏化循环**：
    - 学员在挑战中自然互动（对话、探索、完成小任务）就可以获得积分
    - 积分积累到一定程度，自动解锁初级伴学（如一个基础版旅伴）
    - 想要升级伴学（如从「新手旅伴」→「资深旅伴」，解锁更深度的互动）或获取其他类型的高级伴学，发现积分不够
    - 于是回去做核心动作任务来赚取积分
    - 形成典型的游戏循环：**互动 → 解锁 → 想要更多 → 做任务 → 获得积分 → 继续解锁**
    - 因此高级伴学的设计无法独立于剧本——伴学的接入时机、升级台阶、互动深度变化都必须和叙事节奏咬合
- ⚠️ **Human 必须确认产出**
- 🔲 **待探索**：伴学加入后的互动模式到底长什么样？需要用一个具体主题做原型体验后才能定义

### 环节 4：高级伴学清单设计

- 输入：环节 1 + 3（**依赖环节 3**）
- 产出：可用积分兑换/升级的伴学清单，每种伴学的体验描述、解锁条件、升级路径
- ⚠️ **当前为单机旅程**，不涉及多人合作或社交匹配
- 高级伴学的类型**不限于 NPC**，当前过关的方向：
    - **NPC 类**：向导、旅伴、竞争者、敌人（各有人设与互动方式，可升级）
    - **实战案例包**：真实企业/生活场景的阅读材料，让学员在额外情境中检验自己的判断（不引用任何原作者表述）
    - **隐藏剧情线**：解锁后当前章节出现额外的叙事支线，增加叙事深度和情感投入——如一个看似无关的支线最终和主线汇合，产生 Aha moment
    - **难度调节器**：主动调高挑战难度，获得更高积分倍率——满足高能力学员的挑战欲，同时加速积分积累
    - **复盘视角**：回放自己之前的决策轨迹，带标注地看自己的成长曲线——把「我确实在进步」可视化
    - **情境变体包**：同一个核心动作换一个完全不同的行业/文化/角色背景——检验迁移能力的终极测试
    - 🔲 **待筛选**：具体类型需根据主题和体验目标进一步验证
- ⚠️ **Human 必须确认产出**
- ⚠️ **当前评估：及格但不惊艳**。上述清单中每一项是否都是必要且有效的，尚未论证。需要回答的核心问题：
    - 每种伴学的「不可替代性」在哪里？如果去掉它，学员体验会有实质损失吗？
    - 每种伴学的「有效性假设」是什么？它通过什么机制提升学习效果或爽感？假设能否被验证？
    - 是否存在尚未想到的、真正能让学员「哇」的伴学类型？
    - 🔲 **后续行动**：用一个具体主题跑原型，观察学员对每种伴学的实际反应，再做取舍

### 环节 5：积分系统配置

- 输入：**通用积分框架**（独立于任何主题，一次性设计的科学系统）+ 环节 1 的质量矩阵（含知识类型标签）+ 环节 3 的章节-挑战结构
- 产出：该主题的完整积分参数表
- ⚠️ **Human 必须确认产出**
- **通用积分框架**包含：
    - 分值映射公式：好/中/差 → 分值（如 3 / 2 / 1，具体比例需校准）
    - 复杂度倍率：低 / 中 / 高 → 倍率（如 1.0x / 1.5x / 2.5x）
    - 衰减函数：数学模型 + 知识类型 → 衰减速率映射表（事实性知识衰减快，程序性知识衰减慢）
    - 恢复函数：复习后积分恢复的计算方式（含递增/递减因子）
    - 解锁阈值公式：基于旅程段总点数容量的百分比
    - 目标进度曲线：理想学员应在第几个挑战解锁首个伴学、到达首个里程碑等
- **环节 5 的工作机制**：读取通用框架 + 质量矩阵 + 章节-挑战结构 → 计算旅程总点数容量 → 求解 base_points 使实际进度匹配目标曲线 → 输出完整参数表
- **是否需要 LLM**：如果通用框架已定义完整且环节 1 已标注知识类型，环节 5 是纯算法（参数拟合）；如果需要对维度权重做主观判断，则需 LLM 或人工介入
- ⚠️ **通用积分框架本身需要单独一轮设计**，由学习科学（间隔重复算法 SM-2/FSRS、遗忘曲线研究）+ 游戏设计（进度曲线、经济系统平衡）共同支撑

**以下是具体主题套用时的积分来源和用途**（通用框架的参数在此实例化）：

- **积分来源**：
    - 完成核心动作（按质量分级给分：高质量 > 低质量 > 未完成）
    - 挑战中的自然互动（对话、探索、完成小任务——低分值但持续累积）
- **积分用途**：兑换与升级高级伴学
- **积分衰减/保鲜机制**：
    - 积分随时间衰减——学员长期不回来，已累积的积分逐渐减少
    - 衰减的业务含义：**积分衰减 = 遗忘曲线的游戏化表达**。你不复习，你的「能力积分」就会下降，就像真实的技能会生疏一样
    - 回来做复习任务（在之前已通过的挑战中重新执行核心动作）可以恢复衰减的积分
    - 这把「复习的必要性」从教学设计者的要求变成了学员自己的内在动机——「我的积分在掉，我得回来练」
    - 衰减速率是通用参数，具体主题套用时调整
- ⚠️ **待设计：科学的遗忘曲线模型**
    - 积分衰减要真正对齐遗忘曲线，不能拍脑袋定衰减速率。需要一个科学的衰减函数设计：
        - 衰减曲线的形状（指数衰减？分段线性？基于 Ebbinghaus 的经典曲线还是基于 Leitner 的间隔重复逻辑？）
        - 不同知识类型的衰减速率是否不同（事实性知识衰减快 vs 程序性知识衰减慢？）
        - 复习后的积分恢复量如何计算（完全恢复？部分恢复？恢复量随复习次数递增？）
        - 衰减是否应该有下限（积分不会跌到零，保留「底分」防止学员彻底丧失动力？）
    - 🔲 **后续行动**：这个逻辑很复杂，需要单独做一轮设计，可能需要参考间隔重复算法（SM-2 / FSRS 等）
- 具体的书或路径在设计时「套用」这套积分系统，填入自己的参数（多少分兑换什么、什么质量给多少分、衰减速率），但系统本身的逻辑是通用的

### 学习阶段：学员每次进入旅程时发生什么

#### 旅程的驱动机制：嵌套反馈循环

旅程不是线性序列，而是多层嵌套的反馈循环共同驱动：

```
旅程循环（里程碑为标记，天/周级）
  └── 章节 Chapter：叙事容器，提供背景和情感弧线
        └── 挑战 Challenge：有完成态、可重玩、有积分结算的最小挑战单元
              └── 执行-反馈交互：最小运行单元（一次核心动作执行 + 系统反馈）
```

| 循环层级 | 解决的问题 | 对应的情绪 | 对应的系统功能 |
| --- | --- | --- | --- |
| **执行-反馈交互** | 我这一步做得怎么样？ | 即时满足/挫败 → 调整 | 质量矩阵评估 + 即时反馈 |
| **挑战** | 这个情境我处理得如何？ | 成就感、好奇心 | 积分结算 + 脚手架触发 |
| **章节** | 这段旅程我走到哪了？ | 叙事投入、进度感 | 伴学接入 + 叙事推进 |
| **旅程（里程碑）** | 我整体进步了多少？值得回来吗？ | 成长感、拥有感 | 积分衰减 + 伴学解锁 + 进度可视化 |
- **章节**负责叙事——有起承转合、角色发展、悬念。章节之间是里程碑。
- **挑战**负责游戏性——有明确完成条件、积分结算、可重玩。挑战内可包含多次执行-反馈交互（一个挑战可能需要执行多个核心动作，或同一动作的多次尝试）。
- 判断层在两个粒度上运转：微粒度（单次交互内，按质量矩阵评估并给出即时反馈）和宏粒度（跨挑战/跨章节，判断路径推进、脚手架触发、伴学加载）。

#### 学员的行动空间

以终为始：先穷尽学员在旅程中可能做的所有事情，定义每种行为是否被允许以及系统如何响应。这张表就是判断层的完整决策规则。

| 学员行为 | 是否允许 | 系统响应 |
| --- | --- | --- |
| **执行核心动作，质量达标** | ✅ 主路径 | 按质量分级给积分 → 推进到下一个挑战或更高难度的同一动作 |
| **执行核心动作，质量不达标** | ✅ 允许 | 第 1 次：简短反馈，允许重试
连续 2+ 次不达标：触发脚手架（降低认知负荷的支持）
脚手架后仍不达标：降低当前挑战难度或提供替代路径 |
| **跳过当前挑战** | ✅ 允许，有代价 | 跳过的挑战标记为「未完成」，不给积分
后续挑战如果依赖被跳过的核心动作，难度不降——学员会感受到能力缺口
随时可以回来补做 |
| **回看/重做之前的挑战** | ✅ 鼓励 | 重做可以恢复衰减的积分（复习机制）
重做时如果质量高于首次，给额外奖励积分 |
| **请求兑换高级伴学** | ✅ 积分够即可 | 扣除积分，加载伴学到当前或后续挑战 |
| **与已解锁的 NPC 自然互动** | ✅ 鼓励 | 对话、提问、挑战等互动本身给小额积分（鼓励深度参与而非快速通关） |
| **长时间无操作（同一会话内）** | ✅ 允许 | 超过设定时间 → 推送轻量提示（如 NPC 一句鼓励或一个相关小问题）；不强制 |
| **中途退出又回来** | ✅ 允许 | 断点续传，从上次位置继续
如果间隔较长，积分可能已部分衰减，提示可以先复习 |
| **自由输入/提问/探索** | ✅ 允许 | 与当前场景相关：给予有意义的回应（可能触发隐藏剧情线或额外信息）
偏离主题：温和引导回当前挑战，不惩罚 |
| **主动请求帮助** | ✅ 允许 | 提供引导性帮助（提问式而非直接给答案）；不扣积分
与脚手架的区别：脚手架是系统检测到卡住后自动触发，主动求助是学员发起 |

#### 待论证：路径灵活度问题

上面的行动空间默认学员可以自由输入。但在实际路径设计中，存在一个根本性的张力：

- **现状**：某些路径可能要求学员只能做选择题（从预设选项中选），无法使用自然语言回复，灵活度被限制
- **这样做的目的**：确保学员经历预设的阶段，并反馈系统需要的结构化信号（选了 A 还是 B），从而精确触发下一步
- **反思**：这可能是一个错误的取舍。自然语言的反馈可能更有效——学员用自己的话表达时，认知参与层级更高（ICAP 框架中 Constructive > Active）；同时，LLM 有能力从自然语言中提取系统需要的信号，不必强制结构化输入
- 这涉及的核心权衡是：

|  | 选择题/结构化输入 | 自然语言输入 |
| --- | --- | --- |
| 学员认知参与 | 较低（识别 > 生成） | 较高（必须自行组织表达） |
| 系统信号获取 | 精确、确定 | 需要 LLM 解析，有模糊性 |
| 路径控制 | 强（学员只能走预设分支） | 弱（学员可能说出预设之外的内容） |
| 学习效果（假设） | 可能更低（被动选择） | 可能更高（主动建构） |
| 开发成本 | 低（预设选项 + 分支） | 高（需要鲁棒的 LLM 理解 + 容错） |
- 🔲 **待论证**：是否应该默认允许自然语言输入，只在特定挑战中（如需要精确二选一的决策点）才降级为选择题？需要用原型验证两种模式对学习效果和系统可控性的实际影响

#### 观察 → 判断 → 执行：系统运转方式

上面的行动空间定义了「学员做什么 → 系统做什么」的完整映射。在运行时，这个映射由三个持续运转的业务功能实现：

**观察（眼睛）**：持续记录学员的一切行为，形成实时更新的「学员状态报告」——当前在哪个章节的哪个挑战、已完成哪些核心动作及其质量（按质量矩阵记录）、积分余额、已解锁的伴学、上次访问时间、衰减状态。

**判断（大脑）**：读取状态报告，根据上面的行动空间表做出决策。判断层的决策树就是上表的「系统响应」列。

**执行（手脚）**：把判断层的决策渲染为学员看到的界面——挑战内容、对话、反馈、脚手架、伴学加载、积分变化的可视化。

```
观察（持续记录）→ 判断（查行动空间规则，做决策）→ 执行（呈现内容）
```

## Agent 架构设计

回答每个业务功能由什么样的 Agent 或模块承担，它们之间如何协作。

### 设计阶段的 Agent 架构

设计阶段的用户是 **UMU 的课程设计师**。这个人拿到一个主题，通过与 AI 的对话共创，逐步产出一份旅程蓝图。设计阶段没有组织外的用户，界面尽量**简洁和轻**。

#### 核心模式：Design Copilot + Blueprint + Skills

**一个 Design Copilot（单 LLM Agent）** 贯穿整个设计过程，通过**调用不同的 Skill** 来体现每一步所需的专业能力。

#### 三个组件

**Design Copilot**：一个持续在线的 LLM Agent，是设计师的对话伙伴。

- 它理解整个 Gamecore 框架（核心动作、质量矩阵、章节-挑战结构、伴学、积分），能跨步骤保持上下文
- 当设计师的请求涉及某个环节的专项能力时，Copilot **调用对应的 Skill**——每个 Skill 有专属的 prompt 模板和输出格式约束，确保该环节的产出质量
- Copilot 自身负责：理解设计师意图、决定调用哪个 Skill、将 Skill 的产出写入 Blueprint、回应设计师的追问和修改请求

**Blueprint（旅程蓝图）**：一份结构化数据存储，保存当前设计的完整状态。

- 按步骤组织，每一步的产出都持久化在 Blueprint 中
- 后面的步骤可以读取前面步骤的数据，但不能覆盖（除非设计师显式要求回退修改）
- **每一步都可追溯和回看**——设计师随时可以切换到之前的步骤面板，查看和修改已确认的产出
- Blueprint 各步骤存储的内容：

| 步骤 | Blueprint 中的数据 |
| --- | --- |
| 环节 1 | 核心动作列表、动作之间的关系图、知识类型标签、质量矩阵（含 mini-rubric） |
| 环节 2 | 核心动作 → 体验形式的映射表 |
| 环节 3 | 章节-挑战结构、场景描述、伴学接入点及条件分支 |
| 环节 4 | 伴学清单、persona 骨架、解锁条件、升级路径 |
| 环节 5 | 积分参数表 |

**步骤面板（UI）**：设计师工作界面。

- 布局：**左侧 = 面板**（当前步骤的结构化数据视图），**右侧 = 对话**（与 Copilot 的聊天）
- 面板直接渲染 Blueprint 中当前步骤的数据，表格 / 列表 / 可展开卡片即可
- 所有修改都通过右侧对话驱动——设计师对 Copilot 说"把信息采集的复杂度描述改一下"，Copilot 调用对应 Skill 更新 Blueprint，左侧面板自动刷新
- 设计师可以切换到任意历史步骤的面板回看，也可以在该步骤发起修改（触发回退）
- **回退规则**：修改已确认步骤的产出时，所有依赖该步骤的后续数据标记为「待刷新」，需设计师**显式确认**是否重新生成——不会自动覆盖

#### Skills

每个 Skill 是一组 **prompt 模板 + 输出格式约束 + 校验规则**，由 Copilot 在需要时调用。Skill 不是独立 Agent，它们共享 Copilot 的上下文和 Blueprint 数据。

**Skill: Gamecore 萃取（环节 1）**

- 触发时机：设计师输入主题名称，或要求重新分析核心动作
- 输入：主题名称（+ 设计师的补充说明）
- 输出：核心动作列表 + 动作之间的关系图 + 质量矩阵（含 mini-rubric）→ 写入 Blueprint
- 专业性体现：prompt 模板内置了核心动作的定义标准（指向迁移、可反复执行、≤5个）、质量矩阵的结构要求（评价维度×复杂度×好中差）、知识类型标注要求

**Skill: 体验选型（环节 2）**

- 触发时机：环节 1 确认后，设计师要求选择游戏形式
- 输入：Blueprint 中环节 1 的核心动作清单
- 输出：每个核心动作 → 体验形式的映射 → 写入 Blueprint
- 专业性体现：prompt 模板内置候选体验形式库和选型矩阵（知识类型 × 认知参与层级 → 适配形式），并要求为每个选择给出理由

**Skill: 剧本生成（环节 3）**

- 触发时机：环节 2 确认后，设计师要求生成剧本
- 输入：Blueprint 中环节 1+2 的数据
- 输出：章节-挑战结构 + 叙事骨架 + 伴学接入点的条件分支 → 写入 Blueprint
- 专业性体现：prompt 模板内置章节-挑战结构规范、复杂度递增要求、伴学接入点必须预设「有/无伴学」两版分支等约束
- 这是最重的 Skill，可能分步执行：先生成骨架（章节划分 + 挑战序列），再填充各挑战的场景细节和分支

**Skill: 伴学设计（环节 4）**

- 触发时机：环节 3 有初步结构后，设计师要求设计伴学
- 输入：Blueprint 中环节 1+3 的数据
- 输出：伴学清单 + persona 骨架 + 解锁/升级设计 → 写入 Blueprint
- 专业性体现：prompt 模板内置伴学类型分类、每种伴学需要定义的字段、与剧本接入点的对齐检查
- 🔲 Companion 形式尚未确定，此 Skill 的 prompt 模板待设计

**Skill: 积分配置（环节 5）**

- 触发时机：环节 3+4 确认后，设计师要求配置积分
- 实现方式：**非 LLM**——Copilot 调用一个计算函数，读取通用积分框架 + Blueprint 中的质量矩阵 + 章节-挑战结构，执行参数拟合，输出积分参数表
- 如果拟合结果不合理（如学员太快解锁所有伴学），设计师通过对话调整参数或回退修改前面的步骤
- 如果需要对维度权重做主观判断，退化为 LLM Skill

#### 设计阶段的依赖链与交互流程

```jsx
设计师输入主题
  ↓
[环节1] Copilot 调用「Gamecore 萃取」Skill → 面板展示核心动作+关系图+质量矩阵
  ↕ 设计师通过对话审核和修改
  ↓ ⚠️ 设计师确认
[环节2] Copilot 调用「体验选型」Skill → 面板展示映射表
  ↕ 设计师通过对话审核和修改
  ↓ ⚠️ 设计师确认
[环节3] Copilot 调用「剧本生成」Skill → 面板展示章节-挑战结构
  ↕ 设计师通过对话审核和修改
  ↓ ⚠️ 设计师确认
[环节4] Copilot 调用「伴学设计」Skill → 面板展示伴学清单
  ↕ 设计师通过对话审核和修改（环节3和4可能来回调整）
  ↓ ⚠️ 设计师确认
[环节5] Copilot 调用「积分配置」Skill（计算函数）→ 面板展示积分参数
  ↕ 设计师校验
  ↓ ⚠️ 设计师确认
产出：完整旅程蓝图（Blueprint 的最终状态）
```

#### 为什么是 Skill 而不是独立 Agent

- **上下文连续**：同一个 Copilot 做完环节 1 进入环节 2 时，已经知道所有上下文，不需要跨 Agent 传递数据
- **人在每步共创**：每个环节都有设计师深度参与，不存在 Agent 自主运行的过程。Skill 本质上是 Copilot 在不同环节调用的"专业模式"
- **简洁轻量**：一个 Agent + Skill 模板切换，比 5 个 Agent + Orchestrator + 格式契约简单得多
- **专业性有保障**：每个 Skill 有专属的 prompt 模板（含领域知识、输出格式约束、校验规则），确保即使是同一个 LLM，在不同环节的输出也符合该环节的专业要求

### 学习阶段的 Agent 架构

学习阶段是**实时交互**：学员每说一句话，系统需要在秒级内完成评估、决策、内容生成。核心特征：多角色并行、有状态、需要低延迟。

#### 角色

**State Manager（状态管家）**——代码模块，非 LLM

- 职责：所有确定性计算和状态持久化
- 存储两层数据：
    - **量化层**：积分数值、衰减状态、解锁/升级进度、当前位置（章节/挑战）
    - **质性层**：Judge 每次评估输出的自然语言 evidence（如"学员引用了回避眼神但未考虑外部压力源"），作为历史摘要供后续 Judge 调用使用
- 每次交互的第一步和中间步都经过它：
    - 交互开始时：读取学员状态 → 输出状态快照 + 检测是否有新的 UNLOCK/UPGRADE 事件
    - Judge 评估完成后：接收 grade → 算积分 → 更新状态 → 检测新事件
- 积分计算、衰减公式、阈值比对全部在这里执行，不经过 LLM

**Judge（裁判）**——LLM Agent

- 职责：评估学员表现质量 + 做出路径决策 + 向 Narrator 和 Companion 下达指令
- 输入：
    - State Manager 的状态快照（含 active_companions 列表、事件列表）
    - 当前挑战的上下文 + 该挑战的伴学分支信息
    - 质量矩阵当前复杂度列的 mini-rubric
    - 行动空间规则
    - 学员的输入
    - 历史 evidence 摘要（来自 State Manager 的质性层）
- 输出（结构化）：
    - 各评价维度的 grade（好/中/差）+ evidence（自然语言描述）
    - 路径决策（推进 / 重试 / 触发脚手架 / 切换分支…）
    - 给 Narrator 的指令（方向性的，如"肯定信息采集，追问推理深度"）
    - companion_dispatch：显式路由表，精确指定哪些 Companion 需要响应、各自的指令
    - 如有伴学解锁/升级事件：script_branch 切换指令（激活预设的伴学分支）
- Judge **不产出任何学员可见的文字**。它只打分和下指令。

**Narrator（旁白）**——LLM Agent

- 职责：旅程的主持人，产出所有「旁白视角」的内容——场景描述、反馈、叙事过渡、伴学登场引入
- 输入：
    - 自己的固定 persona
    - 当前章节的叙事上下文 + 当前应走的 script_branch
    - Judge 给它的指令
- 输出：学员看到的旁白文本
- Narrator 的声音从头到尾保持稳定。伴学登场、剧情分支切换等叙事事件都由 Narrator 织入当前场景。

**Companion Agent(s)（伴学）**——每种伴学一个独立 LLM Agent

- 职责：扮演一种特定的伴学角色，与学员互动
- 输入：
    - 自己的 persona（人设、说话风格、行为边界）
    - 与该学员的互动历史摘要
    - Judge 通过 companion_dispatch 路由给自己的指令
- 输出：该伴学 in-character 的回应
- 每种伴学独立的原因：
    - 不同伴学有不同的记忆（旅伴记得友谊，竞争者记得对抗），混在一起会串台
    - 不同伴学有不同的语气和行为规则
    - 加新伴学 = 注册一个新 Agent（提供 persona + 行为规则），不改任何已有代码
- 非 NPC 类伴学（案例库、隐藏剧情线、难度调节器等）也是独立 Agent，但输出形态不同——有的产出文本内容，有的产出结构化参数。Judge 的 companion_dispatch 通过 companion_type 区分路由逻辑。
- 🔲 Companion 的具体形式（哪些类型、每种的输入输出规范）尚未确定，需先完成环节 4 的伴学形式设计

#### 伴学解锁/升级时的剧本同步

伴学解锁不是动态改写剧本，而是**激活设计阶段预设好的条件分支**：

- 环节 3 的剧本在每个伴学接入点都预置了「有伴学版」和「无伴学版」两条路径
- State Manager 检测到积分跨过解锁阈值 → 触发 UNLOCK 事件
- Judge 读取事件 + 检查当前位置是否有该伴学的接入点：
    - 如果在接入点上 → 指令 Narrator 在当前场景引入伴学，同时激活「有伴学版」分支
    - 如果不在接入点上 → 伴学已解锁但延迟到下一个接入点登场，Narrator 可以埋伏笔
- 从激活点开始，Judge 将该 Companion 纳入 active_companions，后续每次交互都评估它是否需要说话
- 升级同理：State Manager 检测升级条件 → Judge 指令 Narrator 叙事过渡 → Companion Agent 切换到高级版 persona

#### 单次交互的完整流程

```jsx
学员输入
  ↓
State Manager（代码）：读取状态 → 输出状态快照 + 事件列表
  ↓
Judge（LLM 调用 #1）：评估 + 决策 + 下达指令
  ↓
State Manager（代码）：grade → 算积分 → 更新状态 → 存储 evidence
  ↓
┌─────────────────────┬───────────────────────────┐
│ Narrator（LLM #2）  │ Companion(s)（LLM #3+）   │
│ 旁白文本            │ 各自 in-character 回应     │
└─────────────────────┴───────────────────────────┘
                      （并行执行）
  ↓
合并渲染 → 学员看到的界面
```

每次交互涉及 2-3 次 LLM 调用：Judge 必须先执行（它的输出驱动后续一切），Narrator 和活跃的 Companion(s) 可以并行。

## 待细化问题与 TODO

以下按阶段列出当前已识别但尚未解决的问题。每一项都需要在推进到具体实现之前回答。

### 业务层面

#### 环节 1：Gamecore 萃取

- 🔲 **质量矩阵的维度如何确定**：当前说"每个动作 2-3 个维度，因动作而异"。需要用一个具体主题（如情境领导力）跑一遍，验证维度选取的逻辑是否可复用
- 🔲 **mini-rubric 的粒度校准**：好/中/差三级是否足够？是否需要更细（如 5 级）？三级描述的边界在实际评估中是否够清晰？

#### 环节 2：游戏体验选型

- 🔲 **体验形式的完整候选库**：当前列举了几种（What-If 模拟器、教会 NPC、悬疑调查、概念法庭），需要系统梳理并定义每种形式的适用条件
- 🔲 **选型规则的可操作性**：知识类型 × 认知参与层级 → 体验形式的映射目前是概念性的，需要变成可执行的选型矩阵

#### 环节 3：剧本与情节生成

- 🔲 **伴学接入点的条件分支到底写到什么粒度**：「有旅伴版」和「无旅伴版」各需要写多详细？是完整的替代文本，还是差异描述 + 共用主干？
- 🔲 **多个伴学同时在场的组合爆炸**：如果旅伴和竞争者同时在场，是否需要单独的分支？组合数量如何控制？
- 🔲 **章节-挑战结构的规模参数**：一个典型旅程有几个章节？每章几个挑战？每个挑战几次执行-反馈交互？需要一个具体主题的原型来锚定

#### 环节 4：高级伴学清单设计

- 🔲 **Companion 形式设计（阻塞性）**：NPC 类伴学之外，案例库、隐藏剧情线、难度调节器、复盘视角、情境变体包各自的交互模式到底长什么样？需要先定义形式，才能设计 Agent 的输入输出
- 🔲 **每种伴学的必要性和有效性论证**：每种伴学用什么机制提升学习效果或爽感？假设能否被验证？需要原型测试
- 🔲 **伴学的升级路径设计**：从 Lv.1 到 Lv.2 到底变什么？persona 切换的幅度多大？新增哪些互动模式？

#### 环节 5：积分系统配置

- 🔲 **通用积分框架的单独设计（阻塞性）**：分值映射、复杂度倍率、衰减函数、恢复函数、解锁阈值公式、目标进度曲线——整套体系需要从学习科学（SM-2/FSRS、遗忘曲线）+ 游戏经济学两个方向做一轮独立设计
- 🔲 **衰减函数的科学依据**：指数衰减还是分段线性？不同知识类型衰减速率差异？复习恢复量公式？衰减下限？
- 🔲 **积分经济的平衡测试**：积分的获取速度和消耗速度是否匹配？学员会不会陷入「积分不够但也不想刷」的死循环？需要数值模拟

#### 学习阶段：业务规则

- 🔲 **路径灵活度论证**：自然语言输入 vs 选择题——需要原型验证两种模式对学习效果和系统可控性的实际影响
- 🔲 **脚手架的评价标准、具体形式与生成归属**：行动空间表中说"触发脚手架"，但脚手架具体长什么样？是降低情境复杂度、给提示问题、还是提供结构模板？需要按核心动作类型分别设计。更关键的问题：脚手架的**评价标准**是什么？效果好坏如何衡量？从评价标准逆向推导**谁来生成脚手架**（Narrator？专门的脚手架模块？Judge 直接指定内容？）
- 🔲 **断点续传的状态恢复策略**：学员离开又回来，系统如何重建上下文？是只恢复数据状态，还是 Narrator 需要做一段"之前的故事"回顾？
- 🔲 **旅程完成态设计**：学员完成所有章节后的旅程回顾和证书。毕业后的复习/持续学习机制需要单独设计（MVP 不处理）

### Agent 架构层面

#### 设计阶段

- 🔲 **各 Skill 的 prompt 模板设计**：每个 Skill 的 prompt 模板需要包含领域知识、输出格式约束、校验规则。需要迭代提示词并用多个主题验证一致性。优先级：环节 1（Gamecore 萃取）> 环节 3（剧本生成）> 环节 2（体验选型）> 环节 4（伴学设计）
- 🔲 **剧本生成 Skill 的分步策略**：这是最重的 Skill，可能需要分步执行（先骨架再填充）或自检循环。单次调用能否产出合格的章节-挑战结构 + 条件分支？
- 🔲 **Blueprint 的数据结构（schema）**：每个步骤在 Blueprint 中存储的具体 JSON 结构需要定义，确保后续步骤能直接读取前序步骤的数据
- 🔲 **步骤面板的 UI 规范**：每个步骤的面板展示什么组件（表格、卡片、树形结构）？
- 🔲 **回退修改的级联规则**：修改已确认步骤时，后续步骤的数据处理策略。当前原则：标记为「待刷新」+ 设计师显式确认。需要定义：哪些修改会触发级联（只改措辞 vs 改核心动作定义）？级联的粒度（整个后续步骤失效 vs 局部失效）？

#### 学习阶段

- 🔲 **Judge 的输出 schema**：当前在聊天中给出了示意性的 JSON，但具体字段（quality 的维度列表、decision 的枚举值、companion_dispatch 的路由字段）需要等业务层的 Companion 形式确定后才能定稿
- 🔲 **Judge 的架构与提示词设计**：Judge 需要同时处理质量评估 + 路径决策 + 脚手架触发 + 多种伴学的路由指令，提示词可能很长。需要探索的方向：是否需要将判断拆成多个 Agent，各负责不同方面（如质量评估、路径决策、脚手架触发）？遵循奥卡姆剃刀——从业务需要出发，只在单一 Judge 确实无法胜任时才拆分。需要测试单 Judge vs 多 Judge 的准确率和延迟差异
- 🔲 **Narrator 的上下文管理**：Narrator 需要知道当前章节的叙事进度、已发生的剧情、活跃伴学。这些上下文如何压缩到 token 预算内？
- 🔲 **Companion Agent 的 persona 规范**：每种伴学的 persona 文档长什么样？包含哪些字段（性格、说话风格、知识边界、与学员的关系阶段、升级后的变化）？需要等 Companion 形式设计完成
- 🔲 **Companion 注册机制的接口定义**：注册一个新伴学需要提供什么？persona 文档 + companion_type + 行为规则？具体的注册接口 schema 待定
- 🔲 **State Manager 的双层存储设计**：量化层的数据结构（积分、位置、解锁状态）和质性层的存储格式（evidence 摘要的长度限制、历史压缩策略）需要具体设计
- 🔲 **合并渲染的排序逻辑**：Narrator 和 Companion 并行输出后，学员看到的内容如何排序？Narrator 在前 Companion 在后？是否需要 Judge 指定顺序？
- 🔲 **延迟预算**：Judge + Narrator + Companion 三次 LLM 调用的总延迟目标是多少？如果超出可接受范围，哪些可以用更快更小的模型？
- 🔲 **Blueprint → Runtime 的数据加载机制**：设计阶段的 Blueprint 最终状态如何被学习阶段的各 Agent 消费？State Manager 需要质量矩阵和积分参数，Judge 需要剧本和伴学分支，Narrator 需要叙事文本，Companion 需要 persona——需要定义从 Blueprint 到各角色输入的映射和加载方式
